1.1

Collection Framework Overview

The Java Collections Framework (JCF) is a unified architecture for representing and manipulating groups of objects. Introduced in Java 2 (JDK 1.2), it provides:

  • Interfaces вҖ” abstract data types (List, Set, Queue, Map, Deque, etc.) that define contracts independent of implementation.
  • Implementations вҖ” concrete classes (ArrayList, HashMap, TreeSet, etc.) that fulfil those contracts with different performance trade-offs.
  • Algorithms вҖ” static utility methods in java.util.Collections (sort, shuffle, binarySearch, min, max, etc.) that operate on any collection.
  • Infrastructure вҖ” Iterator, ListIterator, Comparator, Comparable, Spliterator вҖ” enabling uniform traversal and ordering.

All core interfaces and classes live in java.util. Stream-related extras live in java.util.stream.

🔑 Why Use JCF Over Raw Arrays?
  • Dynamic sizing вҖ” Collections grow/shrink automatically; arrays have fixed length requiring manual resizing and copying.
  • Rich built-in API вҖ” Sorting, searching, shuffling, min/max, frequency counting вҖ” all built in via the Collections utility class and instance methods.
  • Type safety with Generics вҖ” Generics prevent ClassCastException at compile time; arrays are covariant and can cause runtime errors.
  • Interoperability & polymorphism вҖ” Any algorithm coded against Collection<E> works with all implementations; swap ArrayList for LinkedList without changing client code.
  • Iterator & Stream support вҖ” Uniform traversal, lazy pipelines, parallel processing with minimal boilerplate вҖ” impossible with plain arrays.
  • Consistent null and equality semantics вҖ” equals() and hashCode() handled consistently across all implementations.
  • Thread-safe variants ready-made вҖ” Collections.synchronizedList(), CopyOnWriteArrayList, ConcurrentHashMap вҖ” no manual locking needed.
  • Expressive code вҖ” list.removeIf(x -> x < 0) vs manually compacting an array вҖ” JCF leads to shorter, more readable code.

Collection Interface

java.util.Collection<E> is the root interface of the collection hierarchy. It extends java.lang.Iterable<E>, meaning every collection supports the enhanced for loop and provides an iterator().

Collection defines the minimum contract that all collections must honour: adding, removing, querying, and iterating elements. Note: the Collection interface itself does not include index-based methods вҖ” those are defined in List. The table below includes both Collection methods and key List methods (marked List) for a complete everyday reference.

Optional operations (those that may throw UnsupportedOperationException for unmodifiable or fixed-size views) are noted in their descriptions.

📝 Collection vs Collections вҖ” Don't Confuse Them!
  • java.util.Collection<E> (singular) вҖ” an interface. It is the root of the collection hierarchy. Every List, Set, and Queue is a Collection. You declare variables as Collection<String> c = new ArrayList<>();
  • java.util.Collections (plural, with s) вҖ” a utility class with only static methods. It cannot be instantiated (private constructor). Provides algorithms: sort(), binarySearch(), shuffle(), reverse(), min(), max(), frequency(), disjoint(), unmodifiableList(), synchronizedList(), singletonList(), emptyList(), nCopies(), fill(), copy(), rotate(), swap() and more.
  • Similarly: java.util.Arrays is a utility class for working with primitive and object arrays (sort, fill, binarySearch, copyOf, asList, stream).
  • Memory tip: Collection is the blueprint (interface); Collections is the toolbox (utility class).

ArrayList

java.util.ArrayList<E> is the most commonly used List implementation. It stores elements in a contiguous Object[] backing array. When elements are added beyond the current capacity, a larger array is allocated, all elements are copied, and the old array is discarded.

  • Package: java.util
  • Hierarchy: AbstractList вҶ’ AbstractCollection вҶ’ Object; implements List, RandomAccess, Cloneable, Serializable
  • Initial capacity: 10 (default constructor) вҖ” backing array is a shared empty array and expanded lazily on first add().
  • Growth factor: newCapacity = oldCapacity + (oldCapacity >> 1) вүҲ 1.5Г— previous size.
  • Random access: O(1) вҖ” direct index into backing array.
  • Insert/delete at middle: O(n) вҖ” requires shifting elements via System.arraycopy().
  • Append at end: O(1) amortized вҖ” occasional O(n) resize averaged over many adds equals O(1).
  • Permits nulls and duplicates; maintains insertion order.
  • Not synchronized вҖ” use Collections.synchronizedList() or CopyOnWriteArrayList for multi-threaded use.
  • Fail-fast iterators вҖ” throw ConcurrentModificationException if the list is structurally modified during iteration (except through the iterator's own methods).

⚠️ Thread Safety Warning

ArrayList is NOT synchronized. If multiple threads access an ArrayList concurrently and at least one thread modifies it structurally (add, remove, resize), the behavior is undefined вҖ” you may get garbled data, ArrayIndexOutOfBoundsException, or an infinite loop in the worst case.

Additionally, the internal modCount field tracks structural modifications. An iterator captures modCount at creation; if the list is modified outside the iterator during traversal, it throws ConcurrentModificationException on the next iterator call.

Thread-safe alternatives:

  • Collections.synchronizedList(new ArrayList<>()) вҖ” wraps every method in synchronized. Still requires manual synchronization around iteration blocks.
  • CopyOnWriteArrayList<E> (java.util.concurrent) вҖ” each write creates a fresh copy of the backing array. Reads are lock-free and never throw ConcurrentModificationException. Best for read-heavy, rarely-written lists. Write operations are expensive (O(n) per write).
  • Vector<E> (legacy) вҖ” synchronized on every method but coarser-grained; generally avoided in modern code.

LinkedList

LinkedList is a doubly-linked list implementation that implements both the List and Deque interfaces. Each element is stored in a node that holds references to the previous and next nodes, enabling O(1) insert/delete at either end. Random access by index requires traversal, giving O(n) get/set.

  • Part of java.util вҖ” no import beyond java.util.LinkedList needed
  • Not thread-safe вҖ” wrap with Collections.synchronizedList() if needed
  • Allows null elements
  • Can be used as a List, Stack, or Queue/Deque

CopyOnWriteArrayList

CopyOnWriteArrayList is a thread-safe variant of ArrayList from the java.util.concurrent package. The key mechanism: every mutating operation (add, set, remove) creates a brand-new copy of the underlying array, writes to it, then atomically swaps it in. Meanwhile, all readers continue using the old array вҖ” no locking required for reads.

  • Part of java.util.concurrent вҖ” import java.util.concurrent.CopyOnWriteArrayList
  • Reads are lock-free and very fast
  • Writes allocate + copy the entire array вҖ” O(n) per write operation
  • Iterator reflects a point-in-time snapshot вҖ” never throws ConcurrentModificationException
  • Best for read-heavy, write-rare scenarios (e.g., event listener lists)

HashMap

HashMap<K,V> is the most widely used Map implementation. It is backed by a hash table (array of buckets), offers O(1) average time for put/get/remove, and permits one null key and multiple null values. It is not synchronized.

  • Ordering: unordered вҖ” iteration order is not guaranteed.
  • Default initial capacity: 16; default load factor: 0.75.
  • Rehashing doubles capacity when size > capacity Г— loadFactor.

Internal Architecture вҖ” Deep Dive

Step 1 вҖ” Hash function: key.hashCode() is computed, then spread across bits via XOR-shift (h ^ (h >>> 16)) to reduce clustering. Bucket index = hash & (n - 1) (bitwise AND вҖ” works because capacity is always a power of 2).

Step 2 вҖ” Bucket array: An array of Node<K,V>[] (the "table"). Each slot can be null (empty), a single node, a linked list of nodes, or a red-black tree.

Step 3 вҖ” Collision handling:

  • Two keys with the same bucket index form a linked list (chaining).
  • When a chain reaches TREEIFY_THRESHOLD = 8 nodes, it is converted to a Red-Black Tree, reducing worst-case lookup from O(n) to O(log n).
  • If the tree shrinks back to UNTREEIFY_THRESHOLD = 6, it reverts to a linked list.
  • Treeification only happens if capacity >= MIN_TREEIFY_CAPACITY (64); otherwise a resize is triggered first.

Step 4 вҖ” Rehashing: When size > capacity Г— 0.75 the table is doubled and all entries are redistributed. Amortized cost is O(1) per insertion.

// Self-evicting LRU Cache using LinkedHashMap
class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxSize;

    LRUCache(int maxSize) {
        // accessOrder=true: get() moves entry to tail вҶ’ LRU order
        super(maxSize, 0.75f, true);
        this.maxSize = maxSize;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxSize;  // evict head (least recently used)
    }
}

LRUCache<Integer, String> cache = new LRUCache<>(3);
cache.put(1, "one");
cache.put(2, "two");
cache.put(3, "three");
// cache: {1=one, 2=two, 3=three}

cache.get(1);            // access 1 вҶ’ moves to tail
// access order now: 2, 3, 1

cache.put(4, "four");   // evicts eldest = 2 (LRU)
// cache: {3=three, 1=one, 4=four}

System.out.println(cache.containsKey(2)); // false вҖ” evicted!
System.out.println(cache.keySet());       // [3, 1, 4]

WeakHashMap

WeakHashMap<K,V> holds keys using weak references (java.lang.ref.WeakReference). When a key object has no strong or soft references elsewhere in the program, the garbage collector may collect it вҖ” and the corresponding map entry is then automatically removed on the next map operation.

  • Keys are weakly referenced; values are strongly referenced. Values must not hold strong back-references to their keys, or entries will never be cleared.
  • Entry removal is triggered lazily вҖ” map is cleaned on the next structural operation (put, get, size, etc.).
  • Like HashMap: unordered, allows null key and values, not thread-safe.
  • size() may over-report by a small margin until stale entries are expunged.

  • Unordered, not synchronized, permits null keys and values.

ConcurrentHashMap

ConcurrentHashMap<K,V> is a thread-safe, high-performance hash map in java.util.concurrent. It allows concurrent reads and a configurable level of concurrent writes without locking the entire map.

Internal Architecture

Java 7 вҖ” Segment-based Locking

  • Map divided into 16 segments (default concurrency level)
  • Each segment is an independent ReentrantLock + hash table
  • Only the segment containing the target key is locked on write
  • Up to 16 threads can write concurrently

Java 8+ вҖ” CAS + Bucket-level Sync

  • Segments eliminated; array of Node buckets
  • Reads are lock-free (volatile reads)
  • Empty bucket insertion uses CAS (Compare-And-Swap)
  • synchronized only on the first node of a bucket for writes
  • Tree bins (TreeNode) used when bucket length ≥ 8

Comparable vs Comparator

Comparable вҖ” Natural Ordering

java.lang.Comparable<T> defines the natural ordering of a class. A class that implements Comparable is saying: "I know how to compare myself to another object of my type."

  • Defined in the class itself вҖ” modifies the class
  • One ordering per class (single compareTo method)
  • Used automatically by TreeMap, TreeSet, Collections.sort(), Arrays.sort()
  • Contract: compareTo should be consistent with equals
Method Description Time Complexity
compareTo(T o) Returns negative int if this < o; 0 if equal; positive int if this > o O(1) typical
📝 compareTo Contract
  • sgn(x.compareTo(y)) == -sgn(y.compareTo(x))
  • Transitive: x.compareTo(y) > 0 && y.compareTo(z) > 0 implies x.compareTo(z) > 0
  • Consistent with equals (recommended): x.compareTo(y) == 0 should imply x.equals(y)

Comparator вҖ” External/Custom Ordering

java.util.Comparator<T> defines an external ordering for objects, independently of their class. Useful when you need multiple orderings or can't modify the target class.

Method Description Time Complexity
compare(T o1, T o2) Returns negative if o1 < o2; 0 if equal; positive if o1 > o2 O(1) typical
Comparator.comparing(keyExtractor) Creates a Comparator that sorts by a key extracted from each element O(1)
thenComparing(keyExtractor) Secondary sort key applied when primary comparison is equal O(1)
reversed() Returns a Comparator that imposes the reverse ordering O(1)
Comparator.naturalOrder() Returns a Comparator using natural ordering (Comparable) O(1)
Comparator.reverseOrder() Returns a Comparator using reverse natural ordering O(1)
Comparator.comparingInt() Variant for int keys; avoids boxing overhead O(1)
Comparator.comparingDouble() Variant for double keys O(1)
Comparator.comparingLong() Variant for long keys O(1)
thenComparingInt(keyExtractor) Secondary sort for int keys with no boxing O(1)
nullsFirst(comparator) Returns a Comparator that treats null as less than any non-null O(1)
nullsLast(comparator) Returns a Comparator that treats null as greater than any non-null O(1)

Stream API

A Stream is a sequence of elements supporting sequential and parallel aggregate operations. Key properties:

  • Not a data structure вҖ” doesn't store data; processes from a source (collection, array, I/O channel).

Parallel Streams & Primitive Streams

Parallel Streams

A parallel stream splits its source into substreams, processes them concurrently using the ForkJoinPool.commonPool(), and combines the results. Created via collection.parallelStream() or stream.parallel().

Effective When:

  • Large datasets (> few thousand elements)
  • CPU-intensive stateless operations
  • Data is easily splittable (ArrayList > LinkedList)
  • Operations are independent (no shared mutable state)
  • Associative reduction operations (reduce, collect)

Avoid When:

  • Small datasets (thread overhead dominates)
  • I/O-bound operations (threads block, no CPU gain)
  • Stateful / order-dependent operations
  • Operations with side-effects on shared mutable state
  • LinkedList, I/O streams (poor splitability)
JAVA 
// Create parallel stream
long count = IntStream.rangeClosed(1, 1_000_000)
    .parallel()
    .filter(n -> n % 2 == 0)
    .count(); // 500000

// Convert back to sequential
Stream<String> seq = list.parallelStream().sequential();

// forEachOrdered preserves order in parallel
List.of(1,2,3,4,5).parallelStream()
    .forEachOrdered(System.out::println); // always 1,2,3,4,5

// Parallel reduce вҖ” must be associative
int sum = IntStream.rangeClosed(1, 100).parallel()
    .reduce(0, Integer::sum); // 5050
вҡ  Pitfall: Stateful Lambda in Parallel Stream
JAVA 
List<Integer> results = new ArrayList<>(); // NOT thread-safe!
IntStream.rangeClosed(1, 1000).parallel()
    .forEach(results::add); // вӣ” race condition, size unpredictable

// вң… Use collect instead

    .boxed().collect(Collectors.toList());
